17 ## Features
18 
19 There are several implemented features : log levels, targets, thread safety, tasks.
20 
21 ### Log levels
22 <img src="Docs/pics/LogLevel.png" vspace="10" />
23 
24 During development you need a lot of detailed information. Say, while implementing changing some information in database
25 you need to test access to DB, SELECT query and INSERT one. During development you need detailed information about each
26 step, all errors codes, query requests and response etc. When this step is fully implemented and tested you do not need
27 such detailed information. You could remove all this excessive output, but is is better to leave it for future. It will
28 be used in future if you change something in this development.
29 
30 For this goal "logger level" is introduced. You can enable some of them or disable. There is an example of such levels :
31 - "INFO" : general information.
32 - "DEBUG" : detailed data for debugging only.
33 - "WARNING" : some warning about potential problem.
34 - "ERROR" : fail, but only local one. For example, you can't open some file, but application will continue to work.
35 - "CRITICAL" : application can not continue its work.
36 
37 It is an example, you have to introduce your own.
38 
39 On the picture debug and warning messages are disabled. So, they are not sent to output target "CLoggerTxtCOut". But you can
40 enable them in future, so they are prepared, but not output.
41 
42 ### Targets
43 <img src="Docs/pics/Targets.png" vspace="10" />
44 
45 There are different output targets. For now console and file output are implemented, but it is easy to implement another ones.
46 
47 During application work sometimes you need to output the same message to different outputs simultaneously. Say, you can output
48 some important information to the console, detailed information to file and crashes (critical messages) by network.
49 
50 For this goal target entity is introduced. It is implemented by CLoggerGroup class and its children classes :
51 
52 ```cpp
53 Logger::CLoggerTxtGroup<
54  Logger::CLoggerTxtCOut<Logger::CLoggerThrSafe, wchar_t>,
55  Logger::CLoggerTxtFile<Logger::CLoggerThrSafe, wchar_t>
56 > _log;
57 ```
58 
59 In this example `_log` group variable is instantiated. It consists of two logger target - console one and file.
60 
61 You can select different log levels for each target individually. When you send some message to `_log` it will be prepared
62 once and it will be sent for each target.
63 
64 ### Thread safe mode
65 <img src="Docs/pics/MultiThreading.png" vspace="10" />
66 
67 In single thread application one message can be sent to output in the same time. But in multithreading application simultaneous
68 output can create a mess. This is why blocking techniques must be used in this case.
69 
70 In single thread application, however, this overhead is not necessary. This is why CLoggerBase class and its children have
71 boolean thread safe flag as the first template parameter. Set it to false in single thread application and enable it in
72 other cases.
73 
74 ### Task
75 <img src="Docs/pics/Tasks.png" vspace="10" />
76 
77 Multithreading mode has not only mess problem, but a mixture of different logical processes too. Say, you have one thread
78 that works with database, and another one communicating with some hardware device. They have different logical steps
79 output in the same time. It will be difficult to analyze some troubles in messages mixture.
80 
81 Task entity can help you in this situation. "Task" in this logger is the single logical operation without output until its finish.
82 Task operation can be successful or failed. In case of success you don't need any information, but in case of error you
83 need to have the most detailed output.
84 
85 Task collects all messages, but do not outputs them until the finish. Before task destruction you need to set "success" or
86 "fail" status. If the goal was successful, only enabled log levels messages will be output. In the case of fail all messages
87 will be output regardless current log levels mode.
88 
89 You can set some specific log levels for each task if needed.
90 
91 In both cases all messages will be output at the task finish.
92 
93 ### Full conception
94 <img src="Docs/pics/Conception.png" vspace="10" />
95 
96 All those entities can be used simultaneously.
97 
98 For the logger that can be one single target or group ones you set up log levels to be enabled. If needed you can configure
99 individual options for each target.
100 
101 You have to select single thread mode or multithread one for logger.
102 
103 You can start target for some logically connected steps. You prepare all log messages that you need. Before the task finish
104 you have to select ist state, success or fail. In case of fail all messages will be output.
